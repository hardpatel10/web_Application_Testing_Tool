## Phase 4 — Plugin Framework & Plugin SDK — Complete

Builds the complete plugin *architecture* only — no security tool is integrated and no tool is ever executed. New package `backend/plugins/`:

- `models/`: `PluginManifest` (the `plugin.json` schema), `PluginMetadata` (runtime descriptor, always derived via `PluginMetadata.from_manifest`), `PluginHealth`, `PluginConfiguration`, `PluginExecutionContext`/`PluginRawOutput`, `PluginValidationResult`. Reuses `backend.models.enums.TargetType`/`RawOutputFormat` (pure, DB-agnostic `StrEnum`s) rather than duplicating them; only genuinely new concepts (`SupportedPlatform`, `PluginHealthStatus`) get plugin-owned enums.
- `exceptions/`: `PluginLoadError`, `PluginValidationError`, `PluginExecutionError`, `PluginConfigurationError`, `PluginDependencyError`, `PluginNotFoundError` — a framework-internal hierarchy with zero FastAPI/HTTP dependency.
- `core/base_plugin.py`: `BasePlugin` abstract class with all 11 required methods (`metadata`, `health`, `check_installation`, `get_version`, `validate_target`, `prepare`, `build_command`, `execute`, `parse`, `normalize`, `cleanup`).
- `validators/`: three independent passes (structure, manifest, interface) aggregated by a `PluginValidator` facade; used by both the loader (fail-closed) and the `/validate` endpoint (report-only).
- `registry/`: in-memory `PluginRegistry` — duplicate-id rejection, enable/disable, lookup, inter-plugin dependency checking.
- `loader/`: `PluginLoader` discovers every subdirectory of the plugins root and loads it as a synthetic Python package (built via `importlib.util`/`ModuleSpec`, not real `__init__.py` files), so a plugin's `plugin.py` can use ordinary relative imports to reach its `parser.py`/`normalizer.py`/`validator.py` — and so this works regardless of where the configured plugins root actually lives on disk (verified against both the built-in directory and an arbitrary external directory). A broken plugin directory is recorded as a failure and never aborts discovery of the rest ("ignore invalid plugins, produce useful error messages").
- `manager/plugin_manager.py`: `PluginManager` facade (`list_plugins`, `get_plugin`, `reload_plugin`/`reload_all`, `validate_plugin`, `plugin_status`, `installed_plugins`, `check_dependencies`, `enable_plugin`/`disable_plugin`) plus a module-level singleton (`get_plugin_manager`), mirroring the DB engine's singleton pattern rather than `functools.lru_cache` since this is mutable in-memory state.
- `sdk/`: the curated import surface for plugin authors — `BasePlugin` + every model/exception, JSON/XML helpers (XML via `defusedxml`, not stdlib `xml.etree`, since plugins parse tool output that may reflect untrusted target content), file/temp helpers, a namespaced logger factory, and target validators that delegate to `backend.utils.target_validators` (no duplicated validation logic).
- `plugins/example_plugin/`: the one internal reference plugin, implementing every `BasePlugin` method with fixed, non-executing behavior. Never runs a subprocess, never produces a real finding, is never imported by application code — discovered purely by the framework scanning `backend/plugins/plugins/` at runtime.
- `README.md`: full plugin-authoring guide (lifecycle, layout, manifest spec, required methods, validation rules, how to add a real plugin).

**Backend API**: `GET /plugins`, `GET /plugins/{id}`, `GET /plugins/{id}/health`, `GET /plugins/{id}/validate`, `POST /plugins/reload` — via `backend/schemas/plugin.py` + `backend/services/plugin_service.py` (translates `PluginNotFoundError` → `NotFoundError` at the boundary, same pattern as `TargetService`) + `backend/api/routes/plugins.py`. Plugin discovery runs once at app startup (`main.py` lifespan) and logs a summary plus any per-plugin failure.

**Backend tests**: 14 new tests (35 total, up from 21) — API-level tests against the real `example-plugin`, plus framework-only unit tests (loader tolerating missing-files/malformed-manifest/missing-entrypoint directories, registry duplicate-id rejection, not-found, enable/disable, dependency checking, interface validator rejecting an incomplete subclass).

**Frontend**: `Tools` page rewritten into a real Plugin Management page — `PluginCard` component showing metadata, version, health (live-fetched, color-coded badge), enabled status, description, dependencies (with missing ones flagged), and validation status, backed by `usePlugins`/`usePlugin`/`usePluginHealth`/`useReloadPlugins` (TanStack Query) and `plugins-api.ts`. Shows an honest "No plugins installed" empty state when the registry is empty. `npm run build`/`npm run lint` both clean. **No browser automation tool was available this session either** (same limitation as Phase 3) — verified via build/type-check/lint and live network calls to the real backend, not a rendered screenshot.

**Explicitly not done in Phase 4** (per instructions): no Nmap/Nuclei/Nikto/WhatWeb/ffuf/Gobuster/Dirsearch or any other scanner, no subprocess execution, no task queue, no reports, no findings, no correlation. `execute()`/`build_command()` exist on the interface but nothing invokes them against a real tool yet.

## Phase 6 — Assessment Execution Engine — Complete

Builds the complete, tool-agnostic execution engine on top of Phase 4/5's plugin framework. No Nmap or any real scanner is integrated or executed — verified exclusively against a new `dummy-execution` test-fixture plugin (`backend/plugins/plugins/dummy_execution/`, never listed in Tool Management) that only sleeps, logs, and returns success/failure on demand.

- New `backend/workers/` package (planner, queue, manager, state, events, logger, exceptions) — a single bounded pool of `asyncio` workers (a semaphore, never a process pool/Celery/Redis, per this phase's explicit constraint), an `asyncio.PriorityQueue` for ordering/retry-priority/pre-start cancellation, an explicit job-state transition table (`state.py`), and an in-process event bus decoupling the dispatcher from structured per-job logging and the assessment activity log.
- `ExecutionPlanner`: one `ToolExecution` row (job) per (enabled target × requested tool), asking only each plugin's own `validate_target()` — never tool-specific logic. A pairing a plugin can't/won't run becomes a `SKIPPED` job, never a runnable one.
- `ExecutionManager`: the only place that calls a plugin's `prepare()`/`build_command()`/`execute()`/`cleanup()`; persists status/timestamps/output paths as a job progresses; implements cancel (queued or running), retry, and timeout. Reuses `Tool`/`ToolExecution` (Phase 2/5) — no new "Job" table.
- `backend/plugins/sdk/process_runner.py`: real, directly-tested `asyncio.create_subprocess_exec` execution (never `shell=True`) for a *future* tool-integration phase to call — not used by any plugin yet, since `dummy-execution` deliberately never starts an external program and every real tool's `execute()` still refuses.
- 7 new REST endpoints: `POST /assessments/{id}/execute`, `GET /assessments/{id}/progress`, `GET/POST /jobs`, `GET /jobs/{id}`, `GET /jobs/{id}/logs`, `POST /jobs/{id}/cancel`, `POST /jobs/{id}/retry`.
- Migration `a264a27eb29e`: adds `ToolExecution.created_at`/`retry_count`/`status_message` and the new job-lifecycle `ToolExecutionStatus` values.
- 16 new backend tests (66 total, up from 50) exercising the full engine end-to-end over the real HTTP API and a real SQLite DB: planning, concurrency, cancellation (both pre-start and mid-run), retry, timeout, failure (bad exit code and raised exception), live logging, progress reporting, and list/filter.
- **Three real concurrency bugs found and fixed while stress-testing this phase** (see `DECISIONS.md`): a cancellation race that could leave a job stuck at `queued`/`preparing` forever with no safety net; SQLite lock contention (`database is locked`) under the engine's now-genuine concurrent writers, fixed via a busy-timeout and switching to WAL journal mode; and a test-only race between a job's terminal status and the separate, later commit that reverts its assessment's status, fixed by polling for the assessment status in the affected tests instead of reading it once.
- **Frontend**: new `Executions` page — assessment picker, live progress panel, running/queued and history job tables (1.5s polling via TanStack Query, matching every other "live" view in this app), an `ExecuteDialog` (tool/target multi-select), a `JobDetailsDialog` (cancel/retry buttons, gated by the same status sets the backend enforces) with a professional `LogViewer` (monospace, auto-scroll, search, copy, download). `npm run build`/`npm run lint` both clean.
- **Verified live** against the real dev database (not just pytest): created a real assessment/target, ran `dummy-execution` to completion with real captured logs, cancelled a genuinely in-flight job (confirmed via `status: "running"` → `"cancelled"` with `duration` recorded), confirmed the assessment status correctly reverted, retried the cancelled job to a real completion, and confirmed the 409 guard rejecting a cancel on an already-terminal job — then cleaned up the demo assessment. **No browser automation tool was available this session either** (same limitation as every prior phase) — the Executions page was verified via build/type-check/lint and this live API walkthrough, not a rendered screenshot.

**Explicitly not done in Phase 6** (per instructions): no Nmap integration, no real security tool executed, no pause/resume of an in-flight OS process (not portably possible on Windows without unsupported thread-suspension tricks — cancel/kill/retry/timeout are the real, implemented process-control primitives), no findings, no reports, no correlation.

## Phase 8 — Asset Inventory & Observation Engine — Complete

Turns Phase 7's per-job Asset/Service/Observation snapshot into the durable, deduplicated inventory every future scanner plugin normalizes into. Phase 7's `Asset.execution_id` (`nullable=False`) meant every scan run created brand-new rows with zero lookup against prior scans — re-scanning `127.0.0.1` ten times produced ten separate `Asset` rows, only the single best Nmap OS guess was kept (all other candidates discarded), and only the first IPv4/first IPv6 address was kept (dual-stack hosts lost data).

**New domain model** (9 new/restructured tables, `backend/models/`): `Asset` (restructured — `assessment_id` scoping replacing direct `execution_id`, `ipv4`/`ipv6`/`fqdn` split from one `ip_address`, `asset_type`, `fingerprint` merge key, `first_seen`/`last_seen`), `NetworkInterface` (every address, not just the first), `Service` (restructured — `vendor`/`banner`/`fingerprint`/`first_seen`/`last_seen`), `Technology` (extracted from `Service.product`/`.version`), `OperatingSystem` (every OS candidate, not just the best), `Observation` (restructured — `service_id`/`plugin`/`category`/`observation_type`/`fingerprint`; deliberately no severity/confidence), `ObservationEvidence` (append-only, mirrors the existing `FindingEvidence` pattern), `Fingerprint` (protocol-level identity evidence — SSH/TLS/HTTP/SMB/banner/hash — distinct from the merge-key columns), `ExecutionAsset`/`ExecutionObservation` (join tables recording which execution touched which asset/observation, `is_new` distinguishing discovery from re-confirmation). 4 new enums (`AssetType`/`TechnologyCategory`/`ObservationCategory`/`FingerprintType`).

- `backend/services/fingerprinting.py`: pure, deterministic merge-key functions (`asset_fingerprint` — MAC > IPv4/IPv6 > hostname priority; `service_fingerprint`; `observation_fingerprint`), each a namespaced sha256 digest so different identity *kinds* can never collide.
- `backend/services/asset_inventory_service.py`: the merge/upsert engine — the only place a `NormalizedOutput` becomes persisted inventory rows, replacing `ExecutionManager`'s old inline logic. Find-or-insert-or-update per asset/service/technology/OS/observation, keeping the workers package tool-agnostic-dispatch-only.
- `backend/plugins/models/normalized.py` + the Nmap normalizer extended (additively) to pass through *every* OS candidate and *every* address instead of only the best/first — a genuine data-loss fix, verified live (10 OS candidates captured on this dev machine where only 1 survived before).
- One real Alembic migration with actual production data on the line (6 assets/47 services/2 observations from prior sessions' testing) — staged in phases (additive nullable columns → Python-driven backfill computing real fingerprints via the same functions the app uses → tighten to NOT NULL/drop old columns), verified with zero `PRAGMA foreign_key_check` violations and a row-count comparison proving no scans were lost.
- **Real bug found & fixed during downgrade testing**: `Enum(create_constraint=True)`'s CHECK constraint keeps getting re-emitted by Alembic's SQLite batch-mode table-recreate even after an explicit `drop_constraint()` — a narrow, known Alembic/SQLite interaction. Worked around by leaving two enum columns (`asset_type`/`category`) undropped on downgrade (harmless, documented), matching this project's established "upgrade is the fully verified path" precedent from Phases 3/5/7.
- **Real regression found & fixed**: `ExecutionService.get_results` (`GET /jobs/{id}/results`, used by the Phase 7 frontend's `JobResultsView`) queried `Asset.execution_id`, a column Phase 8 removed entirely — caught via live re-verification against a real scan (not assumed safe), fixed to join through `ExecutionAsset` and derive OS name/accuracy from the asset's best current `OperatingSystem` candidate instead of two now-nonexistent columns.
- 5 new read-only API resources (`GET /assets`, `/assets/{id}`, `/services`, `/technologies`, `/observations`, `/operating-systems`, all with search/filter/sort/pagination) plus `GET /search` (unions matches across assets/services/technologies/observations). One consolidated `AssetInventoryQueryService` rather than five near-duplicate services.
- 12 new backend tests (125 total, up from 113) — fingerprint determinism unit tests plus real end-to-end integration tests against actual installed Nmap: single scan, **re-scan the same target** (same `Asset.id`, no duplicate `Service`/`Technology` rows, `last_seen` advances, execution history grows to 2 entries with correct `is_new` flags), observations, search. All passing across 5 consecutive full-suite stress reruns.
- **Frontend**: 5 new list pages (Assets/Services/Technologies/Observations/Operating Systems) plus an Asset Details page (7 tabs: Overview/Network Interfaces/Services/Technologies/Operating System/Observations-with-inline-evidence/Execution History) and a Search page — wired the previously 100%-decorative "Search workspace" header button to a real page instead of a ⌘K command palette (no new dependency). `npm run build`/`npm run lint` both clean.
- **Verified live through the real browser UI** (Playwright): confirmed the 6 pre-existing migrated assets still render correctly, ran a real Nmap OS-detection scan (10 OS candidates captured, all visible in the UI), re-ran a second scan against the same target, and watched the Asset Details Execution History tab correctly show both "Discovered" and "Re-confirmed" entries for the same asset — no duplicate row created.

**Explicitly not done in Phase 8** (per instructions): no Nuclei integration, no correlation engine, no `Finding` rows created or touched (Phase 2's `Finding`/`FindingEvidence`/`FindingReference` remain completely untouched, reserved for a future phase), no severity/CVSS anywhere in the new model, no ⌘K command palette.

## Phase 9 — Correlation Engine & Intelligence Dashboard — Complete

Builds the Correlation Engine (turns Phase 6-8's frozen, real inventory into `Finding` rows via deterministic rules) and rebuilds the Dashboard into a real intelligence dashboard — the first phase to touch Phase 2's `Finding`/`FindingEvidence`/`FindingReference` tables since they were scaffolded. Per this phase's explicit constraint, `backend/workers/` (the execution engine) and the Nmap plugin were never touched; the Correlation Engine is its own, separately-triggered pipeline stage.

**Correlation Engine** (`backend/correlation/`, new top-level package parallel to `backend/plugins`/`backend/workers`): `base.py` (`CorrelationRule` interface: `rule_id`/`title`/`description`/`category`/`severity`/`base_confidence`/`impact`/`remediation`/`references` class attributes + one `evaluate(RuleContext) -> list[FindingCandidate]` method), `models.py` (pure `RuleContext`/`FindingCandidate`/`RuleReference` dataclasses, no DB session), `registry.py` (static in-memory catalog, duplicate-id rejection), `text_utils.py` (keyword matching for NSE-script-output rules). `rules/`: 23 real rules across all 10 required categories (service/technology/os/protocol/configuration/tls/ssh/smb/http/general) — e.g. `SVC-001` Telnet exposed, `SVC-005` database/search service exposed, `TECH-001` end-of-life software, `OS-001` end-of-life OS (best-accuracy candidate only, `>=85%` threshold), `SSH-001` weak SSH algorithms (the phase brief's own worked example), `SMB-001`/`SMB-002` signing/SMBv1, `TLS-001`/`002`/`003` self-signed/expired/weak-cipher, `CONFIG-001` anonymous FTP, `HTTP-001`/`002`/`003` missing headers/directory listing, `GEN-001` version disclosure. Every reference is a real, static CWE/OWASP id (`RuleReference`) — no rule ever attaches a CVE or a CVSS score, since neither can be determined deterministically from collected data without inventing one.

**`backend/services/correlation_service.py`** (write side, mirrors `AssetInventoryService`'s role): per asset, builds a `RuleContext` from its current `Service`/`Technology`/`OperatingSystem`/`Observation` rows, evaluates every registered rule, and — critically — merges *all* candidates one rule returns for one asset into **one** `Finding` (fingerprint = `rule_id` + `asset.fingerprint`, so a rule fires at most once per asset; re-running is idempotent, verified directly). Explicitly **not** wired into `backend/workers/manager.py` — the frozen execution engine is never modified; correlation is triggered explicitly via `POST /correlation/run` (per-assessment or global), recorded as a `CorrelationRun` history row.

## Architectural Change — Linux-Only Runtime

Not a feature phase — a project-direction change applied between Phase 9 and Phase 10 (Nuclei). The application's runtime target is now **Linux only**; Windows is development-only (this repo continues to be edited on Windows via Claude Code, then copied to a Linux machine to run/test after every phase).

**Code audited and changed**: a repo-wide search for Windows-specific runtime logic (`platform.system() == "Windows"`, `.exe` suffixes, `Program Files`/registry/`AppData` lookups, `shell=True`/`cmd.exe`/PowerShell subprocess invocation, hardcoded backslash paths) found exactly one real Windows code path in the entire backend: `backend/plugins/sdk/detection_helpers.py`'s `default_search_directories()`/`find_executable()`, which branched on `platform.system()` to search Windows install locations (`Program Files`, Chocolatey, Scoop, WinGet) and appended a `.exe` suffix. Removed — detection now only ever searches `PATH` (`shutil.which`) plus Linux install directories (`~/go/bin`, `~/.local/bin`, `/usr/local/bin`, `/usr/local/sbin`, `/usr/bin`, `/usr/sbin`, `/opt`, `/snap/bin`). Everywhere else (`core/config.py`, `plugins/sdk/file_helpers.py`, `plugins/sdk/temp_helpers.py`, `plugins/utils/plugin_paths.py`, both Dockerfiles, the frontend) was already pure `pathlib`/POSIX-compatible with no changes needed — a direct result of this project's pre-existing "no hardcoded paths" rule (`.claude/CLAUDE.md`) having been followed from Phase 1 onward. Full backend suite re-run after the change: 153/153 passing.

**Added**: real Linux launcher scripts — `backend/run_backend.sh`, `frontend/run_frontend.sh`, `run_all.sh` (repo root) — mirroring the existing Windows `.ps1`/`.bat` scripts' behavior (create venv/install deps/apply migrations/start server; install node_modules/start Vite) using Bash/POSIX conventions (`set -euo pipefail`, `command -v`, background processes with `nohup`/PIDs instead of separate console windows, since Linux has no equivalent of `Start-Process`/`cmd /k` opening new windows). The Windows scripts are kept as-is (developer-convenience only, per `.claude/CLAUDE.md`'s runtime-platform rules) since the dev machine is still Windows.

**Added**: `docs/LINUX_TOOL_INSTALLATION.md` — apt/dnf/pacman/upstream install instructions for every currently-integrated tool (nmap, plus the 15 detection-only plugins: nuclei, nikto, whatweb, subfinder, sslscan, naabu, katana, httpx, gobuster, ffuf, feroxbuster, dnsx, dirsearch, amass), satisfying the standing rule that a newly integrated tool's Linux installation steps get documented as part of that integration.

**Documentation updated**: `.claude/CLAUDE.md` (new "Runtime Platform" section — Linux-only rules), `.claude/MEMORY.md` (new "Runtime Platform" section + updated "Current Phase"), `.claude/PROJECT_STRUCTURE.md` (notes the new `docs/LINUX_TOOL_INSTALLATION.md` and the `.sh`/`.ps1`/`.bat` script split), this file, and `DECISIONS.md`.

**Explicitly not done**: no new features, no Phase 10/Nuclei work started, no removal of the Windows developer-convenience scripts (still needed since development happens on Windows), no rewrite of already-correct cross-platform code.
